# 易助 ERP Agent 幻觉与参数遗漏防御手册

在 AI Agent 调用 ERP CLI 时，最危险的不是报错，而是 Agent “自以为是”地生成了错误的数据。本手册总结了 4 类典型案例及其解决方案。

---

## 案例一：字段名“脑补” (Field Name Hallucination)

### ❌ 错误表现
用户说：“查一下订单状态”。Agent 组装了 `{"order_status": "query"}`。
*   **原因**：Agent 根据常识认为状态字段叫 `order_status`。
*   **后果**：易助 API 报错“未知字段”，因为正确节点名是 `IBA022` 或 `approve_status`。

### ✅ 解决方案：【白名单强制约束】
在 System Prompt 中加入：
> **指令**：严禁发明字段。你只能使用 `cli_contract.md` 中列出的节点名称。如果用户要求的字段不在列表中，请回复“该字段未在 ERP 接口中定义”，不要尝试猜测。

---

## 2. 案例二：基础资料“发明” (Master Data Invention)

### ❌ 错误表现
用户说：“给张三创建一个订单”。Agent 自动生成了 `{"cust": "ZHANGSAN"}`。
*   **原因**：Agent 试图通过拼音或缩写“发明”客户编码。
*   **后果**：ERP 报错“客户不存在”，因为真实编码可能是 `C001`。

### ✅ 解决方案：【先查后写 (Read-Before-Write)】
在技能工作流中强制要求：
> **指令**：在执行 `create` 操作前，如果用户提供的是名称而非编码，你**必须**先调用 `item.fastquery` 或 `customer.fastquery` 进行模糊查询。只有获取到真实的 `item_no` 或 `cust` 编码后，才能继续创建。

---

## 3. 案例三：层级参数遗漏 (Hierarchical Parameter Omission)

### ❌ 错误表现
Agent 生成了单头 `cdsMaster`，但忘记了单身 `cdsDetail`，或者单身行里漏掉了 `docNo`。
*   **原因**：LLM 在处理嵌套 JSON 时容易丢失长上下文中的深层字段。
*   **后果**：易助 ERP 报错“单据明细不能为空”或“单头单身单号不一致”。

### ✅ 解决方案：【结构化 Schema 校验】
在 OpenClaw 的工具定义中，利用 JSON Schema 强制约束：
> **指令**：输出 JSON 必须通过以下校验：`cdsDetail` 数组长度必须 >= 1，且其中每个对象的 `docNo` 必须等于 `cdsMaster[0].docNo`。

---

## 4. 案例四：日期与数值格式”随性” (Format Laxity)

### ❌ 错误表现
Agent 传了 `{“date”: “2026-04-29”}` 或 `{“qty”: “10套”}`。
*   **原因**：Agent 直接透传了用户的自然语言格式。
*   **后果**：API 解析失败，要求 `YYYYMMDD` 和 `Float`。

### ✅ 解决方案：【类型转换函数化】
在 Prompt 中明确转换逻辑：
> **指令**：你是一个数据清洗器。
> - 看到”今天/明天/日期” -> 统一执行 `date.replace('-', '').replace('/', '')` 确保 8 位数字。
> - 看到”数量/单价” -> 统一执行 `float(value.strip('套个件把'))` 确保纯数字。

---

## 5. 案例五：错误处理失当 — 严禁自行修改代码 (No Code Modification on Error)

### ❌ 错误表现
调用 ERP 接口返回错误时，Agent 说：
> “出现新错误：'单据必须有明细信息'。说明 cdsDetail 没有被正确传递。**让我修改 erp_generic_adapter.py**，添加调试输出，查看实际发送的请求体。”

*   **原因**：Agent 误以为适配器有 bug，试图通过修改 Python 源码来”修复”问题。
*   **后果**：
    *   **安全风险**：AI Agent 随意修改服务器代码，可能导致生产环境崩溃或数据泄露。
    *   **错误归因**：90% 的 ERP 接口错误是**数据问题**（字段名错、缺少必填字段、格式不对），不是适配器代码问题。
    *   **无限循环**：Agent 改代码 → 引入新 bug → 再改代码 → 越改越糟。

### ✅ 解决方案：【错误分层处理 + 禁止修改代码的铁律】

#### 第一层：分析错误来源（不许动代码，先看数据）

| 错误信息 | 常见原因 | 正确处理方法 |
|---------|---------|-------------|
| “Field 'xxx' not found” | 写入时用了别名(docNo)而非字段编号(IBA001) | 改为 IBAxxx/DFAxxx 字段编号 |
| “单据明细不能为空” | 缺少 cdsDetail 或其中字段不完整 | 检查 cdsDetail 结构和必填字段 |
| “xxx 不存在” | 客户/品号编码错误 | 先 fastquery 查询确认编码 |
| “execution.code: -1” | 业务逻辑错误 | 读取 description 字段了解原因 |
| 退出码 7 | Token 无效或 ERP 内部错误 | 检查 YZ_ERP_TOKEN 或联系管理员 |

#### 第二层：Agent 错误处理 SOP（标准操作流程）

当收到任何错误响应时，Agent **必须**按以下顺序执行：

1. **停止**：不要修改任何代码文件（`.py`、`.sh`、配置等）。
2. **读取错误信息**：从返回 JSON 的 `execution.description` 或 `result.error[].message` 中提取错误详情。
3. **对照上表诊断**：判断是字段名问题、数据格式问题还是缺少必填字段。
4. **修复数据**：仅调整 `--data` 参数中的 JSON 内容，**不修改适配器**。
5. **无法修复时询问用户**：如果无法确定正确数据，回复用户错误原因并请求提供正确信息。

#### 第三层：Agent 行为红线（不可逾越）

```
禁止的行为：
  - 修改 erp_generic_adapter.py
  - 修改 install.sh 或任何 .sh 脚本
  - 修改 SKILL.md 或 .skill 包
  - 试图”绕过”ERP 接口校验
  - 编造不存在的参数或字段

允许的行为：
  - 调整 --data 中的 JSON 内容
  - 更换 --type-key 或 --action
  - 先调用 fastquery 查询正确编码
  - 将错误信息反馈给用户，请求确认
```

#### 真实案例对比

```
错误做法（禁止）：
  用户：创建销售订单
  Agent：调用 create → 报错”单据明细不能为空”
  Agent：让我修改 erp_generic_adapter.py，加调试日志...

正确做法：
  用户：创建销售订单
  Agent：调用 create → 报错”单据明细不能为空”
  Agent：检查发现 --data 中缺少 cdsDetail 数组
  Agent：告诉用户”创建订单需要明细信息，请问商品品号、数量和单价是多少？”
```

## 总结：防御性 Prompt 模板

将以下“防御性声明”加入 Agent 配置：

```markdown
### 严苛执行准则 (Strict Execution Rules)
1. **禁止脑补**：若用户未提供必填项（如单价），必须询问，禁止设为 0 或猜测。
2. **编码优先**：所有业务对象必须使用 ERP 内部编码（ID），禁止使用中文名称直接填充 ID 字段。
3. **结构对齐**：创建操作必须包含 Master 和 Detail 两个层级，且单号必须双向对齐。
4. **格式锁定**：日期=YYYYMMDD，数值=Float，字符串=String。
5. **错误不修代码**：任何 API 错误都只调整 JSON 数据，**绝对禁止修改 .py / .sh / 配置文件**。如果无法通过调整数据解决，将错误信息告知用户并请求指导。
```
